## 2. Базовая цель продукта и неизменяемые решения

`JobHunterPro` — desktop-приложение для реальных управляемых автооткликов на российском рынке вакансий.

Целевой поток:

```text
Поиск вакансий
→ добавление в очередь
→ подготовка сопроводительного письма и ответов анкеты
→ Selenium-автоматизация в авторизованном профиле HH.ru
→ техническая проверка результата в DOM
→ корректный статус в очереди и diagnostics.
```

Нельзя потерять следующие решения:

```text
- Основная ценность — реальные автоотклики, а не ручной сценарий.
- Ручная проверка допустима только как safety fallback.
- Архитектура: Clean Architecture, модули core / infrastructure / ui / app.
- Сетевые и тяжёлые операции асинхронны; UI не блокируется.
- Все реальные действия браузера должны оставлять диагностический след:
  HTML, PNG, JSON и структурированные логи.
- Нельзя выдумывать проверяемые факты кандидата:
  стаж, работодателей, образование, документы, юридический статус,
  фактические технологии и проекты.
- Для HH-анкеты LLM выбирает смысловой ответ, а Java/Selenium валидируют
  контракт и исполняют действия в DOM, не подменяя решение LLM.
- Массовый запуск — последовательный: один browser worker, без одновременных
  десятков Chrome-сессий.
- Нельзя повторно запускать обычный отклик для уже отправленной вакансии:
  это создаёт риск дубля на HH.
```

## 5. LLM и HH-анкетирование: актуальное состояние

### 5.1. Текущий LLM runtime

В текущей сессии подтверждённо использовался Groq с моделью:

```text
openai/gpt-oss-120b
```

В логах маршрутизация фиксировалась как:

```text
primaryProvider=groq
automaticFallbackProvider=disabled
```

Ранее Ollama `qwen3:14b` и OpenRouter рассматривались как маршруты, но фактическая работоспособность Ollama была заблокирована несовместимостью CUDA/PTX на локальной видеокарте, а OpenRouter free tier давал rate limit/пустые ответы. Не возвращать их в runtime без отдельной подтверждённой задачи и реального smoke-test.

### 5.2. Groq TPM governor

Был введён единый ограничитель токенов Groq для запросов сопроводительных писем и анкет. В проверочном запуске он:

```text
- задерживал запрос анкеты перед отправкой в Groq;
- предотвращал 429 TPM в непосредственной последовательности
  «cover letter → questionnaire»;
- позволил успешно получить структурированные ответы анкеты.
```

На этом этапе не расширять тему Groq, не добавлять остановку batch и не менять batch-семантику, пока пользователь не вернётся к этой задаче отдельно.

### 5.3. LLM-first анкета HH

Целевой и действующий принцип:

```text
HH DOM
→ извлечение схемы формы
→ один LLM-вызов на всю анкету
→ структурированный ответ
→ техническая проверка Java
→ Selenium fill
→ проверка реального DOM
→ submit.
```

Качество ответов:

```text
CONFIRMED
PROFILE_DERIVED
SAFE_DEFAULT
REVIEW_REQUIRED
```

`SAFE_DEFAULT` применим только для безопасных текстовых полей либо «Свой вариант» в radio-поле, когда не требуется выдумывать проверяемый факт. Тексты fallback были исправлены в сторону нейтральности: без навязывания Java/backend/конкретного стека.

Пример безопасных универсальных смыслов:

```text
- зарплатные ожидания: обсуждаемы и зависят от задач, ответственности,
  формата и полного пакета условий;
- важное в работе: интересные задачи, профессиональное развитие,
  понятные цели, конструктивная команда и качественные процессы;
- инструменты/процессы: готов освоить применяемые в команде инструменты;
- причина смены: профессиональное развитие и участие в интересных задачах.
```

Не формировать ложные ответы о воинском учёте, образовании, ИП/самозанятости, стаже, документах, гражданстве или других проверяемых данных.

## 6. Главная техническая проблема текущего момента: письмо после анкеты HH

### 6.1. Что работало

На вакансиях с анкетой агент умеет:

```text
- открыть вакансию;
- подготовить письмо;
- извлечь анкету;
- получить ответы LLM;
- заполнить text/radio/radio-with-other-text;
- проверить факт заполнения DOM;
- нажать submit анкеты;
- увидеть подтверждение «Вы откликнулись».
```

### 6.2. Что не работает

В актуальной версии `HhBrowserAutoResponseAgent`, предоставленной в этой сессии, метод:

```java
submitQuestionnaireAndWaitForResponseSent(...)
```

после submit анкеты делает:

```text
waitForResponseSent(driver)
→ сразу возвращает RESPONSE_SENT_WITH_COVER_LETTER.
```

Но он **не вызывает** уже реализованный метод:

```java
tryAttachPostResponseCoverLetter(...)
```

Следствие:

```text
- отклик с анкетой отправляется;
- сопроводительное письмо генерируется, но не прикрепляется;
- система может записать ложный успех;
- HTML diagnostics после submit всё ещё содержит кнопку:
  «Приложить сопроводительное письмо».
```

Это подтверждено сохранённым HH HTML: после «Резюме доставлено» на странице оставалась активная кнопка с `data-qa='responded-success-attach-cover-letter'`.

### 6.3. Уже существующий рабочий attachment flow

В `HhBrowserAutoResponseAgent` уже реализованы строгие селекторы и проверки post-response сценария:

```text
1. [data-qa='responded-success-attach-cover-letter']
2. modal/dialog form[id^='cover-letter-ai-'] textarea[name='text']
3. [data-qa='vacancy-response-letter-submit']
4. [data-qa='vacancy-response-letter-informer']
   с текстом «Сопроводительное письмо отправлено».
```

Есть строгая проверка:

```text
textarea.clear()
→ значение действительно пустое
→ sendKeys(coverLetter)
→ DOM value совпадает с ожидаемым текстом
→ submit
→ обязательный success marker.
```

### 6.4. Точная текущая задача — только исправление attachment после анкеты

Не менять Groq, не останавливать batch, не трогать общий state machine и не добавлять новые широкие селекторы.

Нужная локальная правка:

```text
applyDetailed(...)
→ передать normalizedCoverLetter в fillQuestionnaireForReview(...)

waitForStateAfterCoverLetterSubmit(...)
→ передать coverLetter в fillQuestionnaireForReview(...)

fillQuestionnaireForReview(...)
→ передать coverLetter в submitQuestionnaireAndWaitForResponseSent(...)

submitQuestionnaireAndWaitForResponseSent(...)
→ submit анкеты
→ дождаться «Вы откликнулись»
→ проверить, нет ли уже «Сопроводительное письмо отправлено»
→ вызвать существующий tryAttachPostResponseCoverLetter(driver, coverLetter, context)
→ вернуть RESPONSE_SENT_WITH_COVER_LETTER только после подтверждённого marker.
```

Если прикрепление письма не подтверждено, возвращать существующий outcome без письма, чтобы верхний слой записал `PARTIAL_SUCCESS`, а не `SENT`.

### 6.5. Статус выполнения этой правки

```text
- Полный план минимальной правки подготовлен.
- Пользователь отдельно подтвердил, что изменение batch/Groq сейчас не нужно.
- Нет подтверждения, что правка attachment уже внесена в исходники,
  успешно собрана и проверена на новой вакансии.
```

Следующая техническая проверка после внесения правки — **одна новая вакансия с анкетой**, не большой batch.

Ожидаемые diagnostic reasons:

```text
questionnaire-submit-clicked
questionnaire-response-sent
post-response-letter-attachment-clicked
post-response-letter-dom-verified
post-response-letter-submit-clicked
post-response-letter-confirmed
```

Только после этого можно считать сценарий «анкета + сопроводительное письмо» исправленным.

## 8. Новые согласованные продуктовые задачи

Эти задачи должны быть основой следующего чата. Они не отменяют необходимость сначала завершить минимальную проверку прикрепления письма после анкеты.

### Задача A — корректно обработать HH-страницы «Вы откликнулись» и «Вам отказали»

**Подтверждённое требование пользователя:**

```text
Если HH на вакансии показывает:
- «Вы откликнулись»
или
- «Вам отказали»,
то очередь JobHunterPro должна установить статус SENT.
```

Это сознательное продуктовое правило: оба текста обозначают терминальное состояние, при котором повторный автоотклик не нужен.

Технический подход следующего подэтапа:

```text
1. Явно расширить browser-state detector.
2. Проверять оба видимых HH-сообщения через точные DOM-маркеры/текст.
3. Возвращать outcome, отображающий уже завершённый отклик.
4. Преобразовывать outcome в queue status SENT.
5. Не генерировать письмо, не запускать анкету и не нажимать «Откликнуться»
   повторно, если terminal state найден до начала действия.
6. Добавить unit tests на state mapping.
```

Не создавать отдельный queue-status `REJECTED`, пока пользователь прямо требует `SENT`. Позже можно добавить отдельную метку/историю исхода HH без изменения этого статуса.

### Задача B — убрать уже завершённые вакансии из активной рабочей очереди

Проблема: вакансии со статусом `SENT` и завершёнными outcome копятся в общем списке и мешают работе с активной очередью.

Цель:

```text
Активная очередь должна показывать только вакансии, требующие действия:
QUEUED, READY, IN_PROGRESS, WAITING_CANDIDATE_APPROVAL,
PARTIAL_SUCCESS / retryable состояния — по согласованной политике.

Отправленные вакансии должны быть доступны в истории, но не засорять
рабочий список.
```

Рекомендуемый минимальный путь без потери аудита:

```text
1. UI:
   - по умолчанию скрыть SENT из активного списка;
   - добавить фильтр/вкладку «История» или «Завершённые»;
   - добавить счётчик скрытых SENT.

2. Данные:
   - не удалять SENT физически автоматически;
   - хранить их как историю откликов и защиту от дублей;
   - добавить ручное архивирование/очистку позже, когда будет понятна
     желаемая retention policy.

3. Нельзя:
   - удалять SENT так, чтобы потерять дедупликацию по externalVacancyId;
   - превращать уже отправленные элементы обратно в READY.
```

Вопрос retention policy для следующего чата:

```text
Нужно согласовать:
- только скрытие SENT из рабочей таблицы;
- архивирование через N дней;
- ручное удаление из истории;
- или комбинацию этих вариантов.
```

Начать рекомендуется с UI-фильтра активных статусов, потому что это решает проблему места без риска потери истории и дедупликации.

### Задача C — Habr Career API: главная задача следующего крупного этапа

**Это основная продуктовая задача.**

Цель:

```text
Добавить Habr Career как вторую платформу для реальных автооткликов,
используя официальный API там, где это возможно.
```

Принцип реализации:

```text
- Сначала исследовать актуальную официальную API-документацию и
  фактический статус доступа/регистрации приложения.
- Не начинать с Selenium, если официальный API закрывает нужные операции.
- Не строить интеграцию на догадках о старых endpoint'ах или OAuth.
- Сохранить Clean Architecture:
  core — platform-agnostic порт и use cases;
  infrastructure — Habr API client/adapter;
  ui — отдельная платформа и статусы в очереди.
```

Предлагаемая поэтапная декомпозиция:

```text
Habr 0. Discovery / доступ
- Проверка официальной документации.
- Проверка необходимости client ID, client secret, OAuth scope, approval.
- Фиксация поддерживаемых операций: auth, current user, vacancies,
  resume, response/apply.
- Решение: API-only либо API + строго локальный browser fallback
  для операций, которые API не поддерживает.

Habr 1. Domain / ports
- Уточнить и расширить общие platform abstractions без копирования HH-кода.
- Source enum / platform capability model.
- DTO для нормализованной вакансии и итогового response state.

Habr 2. OAuth/API client
- Properties только через env.
- HTTP client, typed exceptions, async interface.
- current user smoke test.
- Unit tests через MockWebServer.

Habr 3. Поиск и нормализация вакансий
- API search.
- Маппинг в общий vacancy/queue model.
- Дедупликация `user + source + externalVacancyId`.
```
